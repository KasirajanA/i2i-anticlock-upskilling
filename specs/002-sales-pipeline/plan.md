# Implementation Plan: Sales Pipeline

## Summary

The Sales Pipeline module enables sales reps to create and move deals through six fixed pipeline stages (Lead In through Closed Won/Lost), with team-scoped visibility, overdue flagging via a daily background job, and a weighted forecast view for managers. The implementation uses FastAPI + SQLAlchemy 2.x async over SQLite WAL for the backend and a React + MUI Kanban board with TanStack Query for the frontend, sharing the auth/notification infrastructure from the core CRM.

---

## Technical Context

**Language/Version**: Python 3.14 (backend), TypeScript 5 / Node 20 LTS (frontend)

**Primary Dependencies**: FastAPI, Pydantic v2, SQLAlchemy 2.x async, aiosqlite, APScheduler 4.x AsyncScheduler, Uvicorn (backend) — React 18, MUI v6, TanStack Query v5, Vite (frontend)

**Storage**: SQLite in WAL mode — single file `crm.db` shared across all CRM modules. WAL enables concurrent reads during the nightly APScheduler write pass.

**Testing**: pytest + pytest-asyncio + httpx.AsyncClient (backend); Vitest + React Testing Library + msw (frontend)

**Target Platform**: Linux server (development and MVP deployment); browser-rendered SPA (frontend)

**Project Type**: REST API service (backend) + single-page application (frontend)

**Performance Goals**:
- SC-001: Pipeline board loads all open deals in under 2 seconds for up to 500 deals per user
- SC-002: Stage change logged immediately — activity log entry visible on same API response
- SC-003: Forecast totals accurate to the cent against source records (zero rounding error)
- API p95 response time ≤ 200ms (constitution NFR)

**Constraints**:
- Cyclomatic complexity ≤ 10 per function (constitution NFR)
- Soft-delete only — no hard deletes anywhere in the system
- Closed Won and Closed Lost are terminal stages — no further transitions permitted
- Backward compatibility with other CRM modules sharing `crm.db` (contacts, auth, notifications)
- No multi-currency, no custom stages, no co-ownership in v1

**Scale/Scope**: MVP — up to ~1 000 deals total, ~500 deals per user visible on board

## Key Design Decisions

### 1. Free-flow Stage Transitions with Terminal Guards

Deals can move between any two non-closed stages without restriction (free-flow), matching the real-world sales reality where deals can skip stages or move backwards. The sole hard constraint — no transitions out of `Closed Won` or `Closed Lost` — is enforced in a single guard in `DealService.change_stage()` rather than a transition matrix, keeping cyclomatic complexity well below the threshold and avoiding premature rigidity for v1.

**Rationale**: A full transition matrix (e.g., a `allowed_transitions` table) is over-engineered for six fixed stages that are not expected to change in v1. The free-flow model is simpler to implement, easier to test (one guard path vs. N×M matrix paths), and fully satisfies the spec requirement.

### 2. is_overdue Flag + APScheduler Daily Job

A persisted `is_overdue` boolean column on the `Deal` table is set by a nightly APScheduler cron job (01:00 UTC) rather than being computed at query time. The flag is cleared immediately in the same DB transaction when a deal advances to a closed stage.

**Rationale**: A computed approach (checking `expected_close_date < today` on every query) requires a function-based index to stay performant and cannot drive notifications without additional machinery. The flag-based approach keeps the board filter query a simple indexed column scan, provides a stable hook for notification dispatch in the scheduler job, and is reset atomically on stage close so there is never a stale overdue flag on a closed deal.

### 3. Weighted Forecast via Hard-coded Probability Map

Forecast weighted values are computed by multiplying each stage's `total_value` by a hard-coded probability constant (`Lead In 10%`, `Qualified 25%`, `Proposal 50%`, `Negotiation 75%`, `Closed Won 100%`, `Closed Lost 0%`). The computation happens in the service layer after a single `GROUP BY stage` SQL aggregate, not in the database.

**Rationale**: Storing probabilities in a DB table adds a configuration management surface that has no value when the stages are fixed and not customisable in v1. Keeping the map in Python (`DealStage.probability()` class method) makes it easy to test in isolation, trivial to extend when custom stages are introduced in a future version, and avoids a JOIN on every forecast request.
